# Glossary

## Структура и роли

| Термин            | Значение                                                                                                                                                                                                                          |
| ----------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **MNZ**           | Monetization — кластер монетизации, продуктовое горизонтальное направление                                                                                                                                                        |
| **Юнит**          | Кросс-функциональная команда (Unit): инженеры, QA, дизайнеры, аналитики, сфокусированная на конкретной потребности пользователя или части сервиса. В контексте кейса — чтобы не путать с unit-тестами, используется «бизнес-юнит» |
| **Кластер**       | Объединение нескольких юнитов, решающих похожие задачи. Бывают продуктовыми и платформенными. В Авито 32 кластера                                                                                                                 |
| **TMG**           | Tech Managers Group — команда руководителей всех кластеров, которая вместе с CTO руководит разработкой                                                                                                                            |
| **Tech Platform** | Платформенный кластер, создающий инструменты для всей разработки                                                                                                                                                                  |
| **Unit Leader**   | Руководитель юнита. Входит в состав кросс-функциональной команды                                                                                                                                                                  |
| **Test Lead**     | M-grade, управление качеством на уровне кластера. Команда 3–7 QA                                                                                                                                                                  |
| **Code Owner**    | Разработчик, владеющий репозиторием. Отвечает за тесты и метрики кода                                                                                                                                                             |

## Процессы

| Термин   | Значение                                                                                                                                                                                                                           |
| -------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **TDR**  | Technical Design Review — ревью технического дизайна или архитектуры приложений и сервисов перед началом реализации                                                                                                                |
| **ZBP**  | Zero Bug Policy — инженерная практика: не копим баги в бэклоге, сразу принимаем решение: чинить сейчас / в очереди / никогда (с обоснованием) / превращаем в задачу если не баг                                                    |
| **ATDD** | Acceptance Test-Driven Development — acceptance criteria определяются до написания кода. В Авито практикуется как «Три амиго»                                                                                                      |
| **SPT**  | Support Project — проект в Jira для обработки багов, крэшей и фичареквестов, поступающих от пользователей через HelpDesk и другие каналы                                                                                           |
| **LSR**  | Live Site Review — процесс разбора инцидентов на работающем сервисе («пожаров на проде») для выявления корневых причин и предотвращения повторения                                                                                 |
| **TMM**  | Team Maturity Model — модель зрелости инженерных команд, представляющая собой список практик (информационная безопасность, качество, перформанс и др.), по которым команды оцениваются и растут. Уровень 2 = обязательный baseline |
| **COP**  | Code Ownership Plugin — плагин, уведомляющий владельца кода об изменениях                                                                                                                                                          |
| **OKR**  | Objectives and Key Results — методология целеполагания для планирования и отслеживания достижений команд                                                                                                                           |

## Метрики и качество

| Термин | Значение | Пример / Детали |
|--------|----------|-----------------|
| **DF** | Deployment Frequency — частота деплоя | Сколько раз в день/неделю катим |
| **LT** | Lead Time for Changes — время от коммита до прода | Включает: code review + сборка + тесты + деплой. Измеряем, чтобы понять, где задержка |
| **MTTR** | Mean Time to Restore — среднее время восстановления после сбоя | От обнаружения до фикса в prod |
| **CFR** | Change Failure Rate — доля неудачных релизов | % релизов, которые привели к инциденту |
| **Defect rate** | Количество багов, доехавших до продакшена | Аналог Defect Leakage |
| **Error rate** | Частота ошибок в работающих сервисах | HTTP 5xx, exception в логах |
| **QS** | Quality Score — не является единым стандартом в Авито. Отдельные компоненты (инциденты, покрытие) оцениваются в рамках TMM и операционных метрик | |
| **SLA** | Service Level Agreement — контрактный уровень сервиса | Юридическое обязательство. Если SLO нарушен — санкции/компенсация |
| **SLI** | Service Level Indicator — что измеряем | Например: latency p99, availability %, error rate |
| **SLO** | Service Level Objective — цель по SLI | Например: availability >= 99.9%, p95 latency < 200ms |
| **Error Budget** | Бюджет ошибок — допустимый объём недоступности | Если SLO = 99.9%, то бюджет = 0.1% времени (~8.7 часов в год). Потратил бюджет — стоп релизам, только стабильность |
| **Apdex** | Application Performance Index — удовлетворённость пользователя скоростью | Формула: (Satisfied + Tolerating/2) / Total. Цель > 0.9. Пример: если 80% запросов < 100ms (satisfied), 10% < 200ms (tolerating), 10% > 200ms (frustrated) → Apdex = (0.8 + 0.1/2) / 1.0 = 0.85 |
| **PBT** | Property-Based Testing — тестирование на основе свойств/инвариантов | Вместо конкретных примеров (input→output) проверяем свойства: «для любого валидного JSON парсер не падает», «сумма подмассивов = сумме массива». Инструменты: fast-check (JS/TS), testing property (Go), Hypothesis (Python) |

## Технологии и инструменты

| Термин | Значение |
|--------|----------|
| **DX** | Developer Experience — подход к созданию удобных и простых продуктов для разработчиков |
| **RPS** | Requests Per Second — запросов в секунду |
| **CI/CD** | Continuous Integration / Continuous Delivery |
| **Quality Gates** | Дополнительные проверки качества, встраиваемые в процесс CI/CD команды |
| **PaaS** | Platform as a Service — внутренняя платформа разработки Avito |
