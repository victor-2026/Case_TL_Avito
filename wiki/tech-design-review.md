# Ревью технического дизайна (TDR)

**Источник:** [avito-tech/playbook/tech_design_review.md](https://github.com/avito-tech/playbook/blob/master/tech_design_review.md)

## Что такое TDR

Technical Design Review — процесс ревью технического дизайна перед реализацией. Цель — выявить проблемы архитектуры на ранней стадии, когда их исправление дешевле всего.

## Для кого и каких проектов

- Микросервис, который пересекается с другими командами
- Изменения, влияющие на другие юниты
- Проекты под риск SLO/SLA

## Входные требования

- Документ с описанием дизайна (формат: .md в репозитории)
- Описание API, схемы данных, архитектуры
- Trade-offs и альтернативы
- План тестирования и мониторинга

## Процесс

1. Автор создаёт документ TDR
2. Назначает ревьюверов (Tech Lead смежного юнита, эксперт)
3. Ревьюверы комментируют в течение约定的 времени
4. Автор либо принимает, либо отвечает
5. После одобрения — реализация

## TDR для тестирования

TDR применим к стратегии тестирования. Документ включает:
- Какие тесты пишем: unit / API / UI / E2E
- Покрытие: какие сценарии, какие кейсы не покрываем (и почему)
- Тестовые данные: как готовим, где храним
- CI/CD: как тесты запускаются
- Метрики тестирования
