# Соседское соглашение

**Источник:** [avito-tech/playbook/neighborhood-agreement.md](https://github.com/avito-tech/playbook/blob/master/neighborhood-agreement.md)

**Ключевой документ для кейса.** Регулирует взаимодействие между юнитами при внесении изменений в чужой функционал.

## Цель
Снизить коммуникационные издержки и сократить баги от внесения изменений юнитами в функционал друг друга.

## Краткое руководство (11 правил)

| Правило | Руководство к действию |
|---------|----------------------|
| Правило бойскаута | Оставь код репозитория лучше, чем он был до твоего прихода |
| Куда хочешь, туда и пушь | PR можно в любой репозиторий, если не запрещено политиками безопасности |
| В чужой код со своим уставом не лезь | Используй Coding Standard команды-владельца кода |
| Запушил в чужой код, не мёржь без согласия владельца | Следуем политике Code Review команды-владельца |
| Написал новый код, напиши и тесты | Тесты — часть документации |
| Написал код, а тесты не упали? Проверь и исправь тесты | Баг должен ломать тесты. Исправляет CODE OWNER |
| Кто сломал, тот и чинит | Автотесты, метрики, логирование |
| Отклоняешь на ревью, аргументируй | Нельзя отклонить PR без аргументов |
| Пришел тикет на ревью? Проведи сегодня | Code Review не позже следующего дня |
| Если сломали твой код, а ты не узнал — сам виноват | Используешь соседнее API — напиши тест, который проверит, что твой юнит не сломан |
| Владеешь кодом — отвечаешь за его тесты и метрики | Оунишь репозиторий — покрой тестами, метриками и логированием |

## Правила внесения изменений в чужой функционал

Основное правило — **«Keep Going Green»**. Нормальное состояние билдов и тестов — зеленое.

- Получить одобрение владельца функционала до изменений
- Поддерживать работоспособность мониторинга, метрик и авто-тестов владельца
- Все авто-тесты юнита поддерживаются в актуальном состоянии
- Утративший актуальность тест удаляется по согласованию с владельцем
- При расширении функционала покрытие не снижается
- Чьё изменение завалило тесты на staging — тот их и чинит
- Юнит-владелец авто-теста информируется через **COP (Code Ownership Plugin)**

## Ответственность владельца

- Поддерживать покрытие кода тестами **в районе 50%-80%**
- Если баг обнаружен через несколько релизов — исправляет владелец
- Юнит должен понимать, что проверяет каждый тест: DB-tests, Unit-tests, API-tests, UI-tests
- Если старые тесты были зеленые, а новые правки их сломали — автор правок исправляет
- Владелец обязан поддерживать тестовые среды (Staging)
- Ответственность за сохранение функционала — на том, кто последним его правил

## Меры контроля качества автоматизации тестирования

- Команда QA кластера выполняет ревью тестов с произвольной периодичностью
- Соблюдаются общие требования к авто-тестам
- Проводится ревью авто-тестов для поддержания порядка
- Разрабатываются рекомендации по автоматизации тестирования
