# Test Lead MNZ — Case Presentation

**Роль:** Test Lead, кластер MNZ (M-grade)
**Уровень:** M1–M2 (менеджерский трек)
**Сценарий:** Конкретная проблемная ситуация в одном из юнитов кластера

---

## Context

### О роли

Test Lead в MNZ — связующее звено между бизнес-задачами кластера и инженерными стандартами компании.

- **Уровень кластера:** Отвечаю за организацию качества на уровне всего MNZ (back-end, web, iOS, Android)
- **Команда:** Управляю 3–7 QA-инженерами, распределёнными по юнитам кластера
- **Связь с платформой:** Внедряю стандарты (TMM, пирамида тестирования, Quality Gates), сформулированные командой Quality (Tech Platform)
- **DS-направление:** В кластере есть DS-команды (алгоритмы ликвидности, аукцион) — качество их моделей также в моей зоне

### Конкретный сценарий (из кейса)

В одном из юнитов моего кластера сложилась следующая ситуация:

| | Команда 1 | Команда 2 | Команда 3 (соседний юнит) |
|---|---|---|---|
| **Что делает** | Back-end агрегатор, отдаёт данные для команды 2 | Рекламный блок на Карточке (web + mobile + back-end) | Платформа «Карточка объявлений» |
| **QA** | Нет | 1 QA | Неизвестно |
| **Проблема** | Багов почти нет — почему? | Много багов, жалобы саппорта и клиентов | Качество неизвестно |

## Q1: С чего начну исследование?

### Первые 2 недели — сбор данных

**Команда 1 (back-end, без QA, 0 багов)**

Гипотезы:
- **H1:** Код действительно качественный — хорошее unit-покрытие, code review, CI gates
- **H2:** Баги есть, но их не регистрируют (чинят в моменте, без трекинга)
- **H3:** Баги есть, но их не находят — нет исследовательского тестирования

Действия:
1. Coverage отчёты (unit + API) — какой процент кода под тестами
2. CI/CD pipeline — есть ли Quality Gates? Что ловит до прода?
3. Интервью с dev: как тестируют изменения? Как проходит code review?
4. SPT-дашборд — есть ли обращения на этот сервис?
5. Review бэклога: только feature tasks или были баги, которые закрыли без формального трекинга?

**Команда 2 (UI + back-end, 1 QA, много багов)**

Это источник боли. Без детальных данных любые решения — гадание.

Действия:
1. **Классификация бэклога:** распределить все открытые баги по категориям (UI / API / data / integration / unknown)
2. **Severity-анализ:** сколько P0–P1 в месяц, как быстро их чинят (MTTR)
3. **Root cause:** разобрать 5 крупнейших инцидентов за последний квартал
4. **Оценка QA:** что делает текущий QA? manual only? автотесты? регрессия?
5. **SPT-дашборд:** жалобы клиентов — это одни и те же проблемы или разные?
6. **Отделить баги команды 2 от багов платформы:** проблема в коде команды или в интеграции с Карточкой?

**Команда 3 (соседний юнит, платформа «Карточка»)**

Гипотеза: часть багов команды 2 — следствие нестабильности или breaking changes платформы.

Действия:
1. Встреча с tech lead / QA команды 3 — узнать их SLI/SLO, как уведомляют о changes
2. Проверить, существует ли **Соседское соглашение** между юнитами. Если нет — инициировать его заключение
3. Попросить доступ к coverage / incident rate дашбордам команды 3
4. Проверить контракты API: менялись ли недавно? Были ли breaking changes?
