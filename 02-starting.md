# Test Lead MNZ — Case Presentation

**Сценарий:** Конкретная проблемная ситуация в одном из юнитов кластера

---

## Context

### О роли

- **Уровень ответственности тест-лида в кейсе отличается от вакансии:** Отвечает за организацию качества на уровне нескольких команд одного бизнес-юнита (back-end, web, iOS, Android)
- **Роль тест-лид:** Управляет 3–7 QA-инженерами, распределёнными по бизнес-юнитам кластера
- **Связь с командой/платформой Quality:** Внедряет стандарты (TMM, пирамида тестирования, Quality Gates, ATDD,...), фреймворки, инструменты и метрики сформулированные командой Quality (Tech Platform)


### Конкретный сценарий (из кейса)

В одном из бизнес-юнитов сложилась следующая ситуация:

|                | Команда 1                                       | Команда 2                                                          | Команда 3 (соседний юнит)       |
| -------------- | ----------------------------------------------- | ------------------------------------------------------------------ | ------------------------------- |
| **Что делает** | Back-end агрегатор, отдаёт данные для команды 2 | (Рекламный?) блок на Карточке объявлений (web + mobile + back-end) | Платформа «Карточка объявлений» |
| **QA**         | Нет - почему?                                   | 1 QA                                                               | Неизвестно                      |
| **Проблема**   | Багов почти нет — почему?                       | Много багов, жалобы саппорта и клиентов - ZBP?                     | Качество неизвестно             |

## Q1: С чего начну исследование?

### Первые 2 недели — сбор данных

Факты:
	**Команда 1 (back-end, без QA, 0 багов)**
	[[! Если в команде нет QA-инженера]]

Гипотезы:
- **H1:** Код действительно качественный — хорошее покрытие модулей функциональными и API-тестами, code review, CI gates
- **H2:** Баги есть, но их не регистрируют (чинят в моменте, без трекинга)
- **H3:** Баги есть, но их не находят — нет исследовательского тестирования, нет багов от соседней команды

Действия:
1. Coverage отчёты (unit + API) — какой процент кода под тестами
2. Если там нет QA - то есть ли тестирование контрактов? Кто его проводит?
3. Проверить контракты API: менялись ли недавно? Были ли breaking changes?
4. CI/CD pipeline — есть ли Quality Gates? Что ловит до прода?
5. Интервью с dev: как тестируют изменения? Как проходит code review?
6. SPT-дашборд — есть ли обращения на этот сервис? Наверное нет -  не клиент-сайд
7. Review бэклога: только feature tasks или были баги, которые закрыли без формального трекинга?
8. Как происходит передача и документирование в команду 2 ?

**Команда 2 (UI + мобайл + back-end, 1 QA, много багов)**

Это источник боли. Без детальных данных любые решения — гадание.
Большой список багов в бэклоге - несмотря на политику и практику ZBP.


Действия:
1. **Классификация бэклога:** распределить все открытые баги по категориям (UI / API / data / integration / unknown)
2. **Severity-анализ:** сколько P0–P1 в месяц, как быстро их чинят (MTTR)
3. **Root cause:** разобрать 5 крупнейших инцидентов за последний квартал
4. **Оценка QA:** что делает текущий QA? manual only? автотесты? регрессия?
5. **SPT-дашборд:** жалобы клиентов — это одни и те же проблемы или разные? Это уже указано в SPT - по частоте и приоритетам/критичности. 
6. Процесс Live Site Review (LSR) = здесь происходит кластеризация по категориям  (UI / API / data / integration / unknown)
7. После LSR возможно инициировать TDR
8. **Отделить баги команды 2 от багов сервиса команды 1 и платформы:** проблема в коде команды или в сервисе-агрегаторе или интеграции с Карточкой?

**Команда 3 (соседний юнит, платформа «Карточка объявления»)**

Гипотеза: часть багов команды 2 — следствие нестабильности или breaking changes платформы.

Действия:
1. Встреча с tech lead / тест-лидом / QA команды 3 — узнать их SLI/SLO, как уведомляют о changes
2. Проверить, существует ли **Соседское соглашение** между юнитами. Если нет — инициировать его заключение
3. Попросить доступ к coverage / incident rate дашбордам команды 3
4. Проверить контракты API: менялись ли недавно? Были ли breaking changes?
